# TripMate Multilingual Support (i18n) — Design Spec

## Overview

Add multilingual support for Portuguese (pt-BR), English (en), and Spanish (es) across the entire app. A language switcher on ProfilePage lets users choose their language; the selection is persisted to `localStorage`. On first visit, the browser's language is auto-detected and mapped to one of the three supported locales, falling back to `pt-BR` if no match.

**Library:** `react-i18next` + `i18next` + `i18next-browser-languagedetector`

> **Note on PlansModal source language:** `PlansModal.tsx` currently has hardcoded Portuguese strings (e.g. `"sempre grátis"`, `"/ano"`, `"/mês"`, feature bullets in Portuguese). When adding i18n keys to that component, replace ALL hardcoded strings — both the Portuguese ones and the already-English ones — with `t()` calls.

> **Note on browser language detection:** `navigator.language` may report `"pt"` (without region). i18next will attempt to match against `supportedLngs: ['en', 'pt-BR', 'es']` and fail, then fall through to `fallbackLng: 'pt-BR'`. This is the intended behaviour — Portuguese speakers without a region code get Portuguese. Do not add a `"pt"` entry to `supportedLngs` as it would create an empty resource set.

## Supported Languages

| Code | Label | Flag |
|------|-------|------|
| `pt-BR` | Português | 🇧🇷 |
| `en` | English | 🇺🇸 |
| `es` | Español | 🇪🇸 |

**Default:** Auto-detected from `navigator.language`. If the browser language is not one of the three (including `"pt"` without region), falls back to `pt-BR`.

**Persistence:** `i18next-browser-languagedetector` reads/writes `localStorage` key `i18nextLng` automatically.

## Architecture

```
src/
├── i18n/
│   ├── index.ts              ← i18next config (init, detector, resources)
│   └── locales/
│       ├── en.json           ← English strings
│       ├── pt-BR.json        ← Portuguese strings
│       └── es.json           ← Spanish strings
├── components/
│   └── LanguageSwitcher.tsx  ← 3-button row: PT · EN · ES
└── pages/
    └── ProfilePage.tsx       ← add LanguageSwitcher alongside DarkModeToggle
```

`src/main.tsx` imports `src/i18n/index.ts` before rendering the app.

All components call `useTranslation()` from `react-i18next` and replace hardcoded strings with `t('key')`.

**Global test mock:** `src/test-setup.ts` currently contains only `import '@testing-library/jest-dom'`. The implementer MUST add the following mock as the first task, before touching any component:

```ts
vi.mock('react-i18next', () => ({
  useTranslation: () => ({
    t: (key: string, opts?: Record<string, unknown>) => {
      if (opts && typeof opts === 'object') {
        return Object.entries(opts).reduce(
          (s, [k, v]) => s.replace(new RegExp(`{{${k}}}`, 'g'), String(v)),
          key
        )
      }
      return key
    },
    i18n: { changeLanguage: vi.fn(), language: 'en' },
  }),
  initReactI18next: { type: '3rdParty', init: () => {} },
}))
```

This mock returns the key itself (with interpolation applied) so existing snapshot / text assertions will need updating to match the key string rather than the translated value. Tests that check for specific visible text (e.g. `"Title is required"`) should be updated to check for `"event.titleRequired"`, OR the test file can override the mock to return the English string.

## Components

### `src/i18n/index.ts` — new

```ts
import i18n from 'i18next'
import { initReactI18next } from 'react-i18next'
import LanguageDetector from 'i18next-browser-languagedetector'
import en from './locales/en.json'
import ptBR from './locales/pt-BR.json'
import es from './locales/es.json'

i18n
  .use(LanguageDetector)
  .use(initReactI18next)
  .init({
    resources: {
      en: { translation: en },
      'pt-BR': { translation: ptBR },
      es: { translation: es },
    },
    fallbackLng: 'pt-BR',
    supportedLngs: ['en', 'pt-BR', 'es'],
    detection: {
      order: ['localStorage', 'navigator'],
      caches: ['localStorage'],
      lookupLocalStorage: 'i18nextLng',
    },
    interpolation: { escapeValue: false },
  })

export default i18n
```

### `src/components/LanguageSwitcher.tsx` — new

A compact 3-button row. Active language gets a blue highlight (same `bg-primary` as the app's primary color). No icon — just text labels (PT, EN, ES).

```tsx
import { useTranslation } from 'react-i18next'
import { cn } from '@/lib/utils'

const LANGUAGES = [
  { code: 'pt-BR', label: 'PT' },
  { code: 'en',   label: 'EN' },
  { code: 'es',   label: 'ES' },
]

export default function LanguageSwitcher() {
  const { i18n } = useTranslation()
  return (
    <div className="flex gap-1" role="group" aria-label="Language">
      {LANGUAGES.map(({ code, label }) => (
        <button
          key={code}
          onClick={() => i18n.changeLanguage(code)}
          aria-pressed={i18n.language === code}
          className={cn(
            'px-2.5 py-1 rounded-lg text-xs font-semibold transition-colors',
            i18n.language === code
              ? 'bg-primary text-white'
              : 'text-muted hover:text-foreground'
          )}
        >
          {label}
        </button>
      ))}
    </div>
  )
}
```

## Data Flow

```
i18n init (main.tsx import)
  └── LanguageDetector reads localStorage('i18nextLng')
      └── if missing: reads navigator.language → maps to supported locale
          └── 'pt' and any unknown → fallbackLng 'pt-BR'

User clicks language button (LanguageSwitcher)
  └── i18n.changeLanguage(code)
      └── LanguageDetector writes localStorage('i18nextLng')
          └── all useTranslation() hooks re-render with new strings
```

## Gap Warning Internationalization

> **Current state:** `gapDetection.ts` returns `message: string`; `useTimelineGroups.ts` carries `message` on the gap `RenderItem`; `GapWarningCard` accepts `message: string`; `TripDetailPage` and `MapView` pass `message={item.message}`. All 7 files listed below must change as part of this feature.

`detectGaps()` in `src/lib/gapDetection.ts` currently returns `message: string` (hardcoded English). This must be refactored so pure functions do not depend on i18n. The `GapWarning` interface changes:

```ts
export interface GapWarning {
  afterEventId: string
  beforeEventId: string
  fromTitle: string   // replaces message — title of the first accommodation
  toTitle: string     // replaces message — title of the second accommodation
}
```

The full propagation chain for this change:

1. **`gapDetection.ts`** — drop `message`, build `fromTitle`/`toTitle` from event titles
2. **`gapDetection.test.ts`** — update all gap assertions from `gaps[0].message` to `gaps[0].fromTitle` / `gaps[0].toTitle`
3. **`useTimelineGroups.ts`** — update `RenderItem` gap variant: replace `message: string` with `fromTitle: string; toTitle: string`; update the push call to forward `fromTitle: gap.fromTitle, toTitle: gap.toTitle`
4. **`useTimelineGroups.test.ts`** — update gap literal objects from `{ message: '...' }` to `{ fromTitle: '...', toTitle: '...' }`
5. **`TripDetailPage.tsx`** — update `<GapWarningCard>` call from `message={item.message}` to `fromTitle={item.fromTitle} toTitle={item.toTitle}`
6. **`GapWarningCard.tsx`** — accept `fromTitle`/`toTitle` props, call `t('trip.gapWarning', { from: fromTitle, to: toTitle })`
7. **`MapView.tsx`** — if it renders `<GapWarningCard>`, update its call site the same way as TripDetailPage
